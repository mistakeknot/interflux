# Track A — Architecture & Design Review

## Overall Assessment

interflux is a markdown-and-script orchestration engine whose *core* decomposition is sound: commands are thin shims over skills, skills delegate phase logic to `phases/*.md` files read on demand, and the two skills compose cleanly (`flux-review-engine` fans out to N independent `flux-engine` runs rather than reimplementing dispatch — a genuine wrapper, not a god-skill). The agent/command/skill/config boundaries are real and mostly respected. The dominant architectural problem is **not coupling but entropy and drift**: the canonical documentation (`AGENTS.md`, `CLAUDE.md`, `agents/architecture.md`) describes a plugin that no longer exists on disk (1 skill vs 2, 1 MCP server vs 2, `skills/flux-drive/` vs `skills/flux-engine/`), the SKILL-compact caching mechanism is stale and orphaned, several cross-document contract links point at files that do not exist, and the orchestration phase files (`launch.md`, `synthesize.md`) have accreted ~15 optional progressive-enhancement integrations each, pushing a single file toward god-module territory. The bones are good; the connective tissue has rotted faster than the docs were updated. Most fixes are low-risk: re-sync docs to disk, regenerate or delete the compact artifact, fix or remove dangling links, and extract the optional-integration steps out of the hot path.

---

- **[P1] Canonical docs describe a structure that no longer exists on disk** — `AGENTS.md:17,216,229`, `CLAUDE.md:7,20`, `agents/architecture.md:8` — The development guide and overview claim "1 skill (unified flux-drive)", "1 MCP server (exa)", and reference paths under `skills/flux-drive/`. On disk there are **2 skills** (`skills/flux-engine/`, `skills/flux-review-engine/`), **2 MCP servers** (`exa`, `openrouter-dispatch` per `plugin.json`), and the skill directory was renamed to `flux-engine`. `AGENTS.md:216` and `:229` send a reader to `skills/flux-drive/phases/` and `.../references/` which do not exist. For a plugin whose entire substance is its prose, stale wayfinding docs are an architecture defect: contributors and the agents themselves navigate by these paths. The CLAUDE.md validation one-liners are internally contradictory too — line 20 says "Should be 2" with a comment explaining the second skill, but the same file's overview (line 7) still says "1 skill". Smallest fix: update AGENTS.md Quick Reference + Architecture tree, CLAUDE.md overview, and `agents/architecture.md` to the real two-skill / two-MCP / `flux-engine` layout in one pass.

- **[P1] SKILL-compact.md is a stale, orphaned parallel copy of the orchestration logic** — `skills/flux-engine/SKILL-compact.md`, `skills/flux-engine/.skill-compact-manifest.json` — A 16KB second copy of the full SKILL.md algorithm exists alongside a manifest of SHA-256 hashes meant to detect drift. The manifest records `SKILL.md` as `543f82…` but the actual file now hashes to `80983b…` — the compact copy is **out of sync with its source** and the drift-detection it was built to enforce has already failed silently. Worse, nothing in the runtime references it: `grep` finds SKILL-compact only in old `docs/plans/` and `docs/research/`, never in any phase file, command, script, or hook. This is a classic parallel-architecture hazard (CLAUDE.md's own fd-architecture brief warns against "hidden feature flags creating parallel architectures"): a second source of truth that can diverge from the primary and mislead anyone who reads it. Smallest fix: either delete `SKILL-compact.md` + `.skill-compact-manifest.json` if the compaction feature was abandoned, or regenerate it and wire a pre-commit check that fails when the manifest hash mismatches. Deletion is the lower-risk default given there is no live consumer.

- **[P1] Dangling cross-repo contract links escape the repo and target nonexistent files** — `skills/flux-engine/SKILL.md:196,256` — Step 1.2a.0 and the triage-persistence step link to `[routing-contract.md](../../../../../docs/contracts/routing-contract.md)` and `[triage-log-schema.md](../../../../../docs/contracts/triage-log-schema.md)`. The `../` count resolves to `../../../docs/contracts/...` *above the interflux repo root*, and `docs/contracts/` does not exist anywhere in the repo (`find` confirms neither file is present). These are load-bearing contracts — the schema-version fail-open logic (line 198) and the triage.jsonl schema (line 256) both defer their definitions to files an implementer cannot open. Either the contracts were never migrated when the skill moved, or the link depth is wrong. Smallest fix: create `docs/contracts/{routing-contract,triage-log-schema}.md` (even as stubs documenting the inline schema), or inline the schema into SKILL.md and drop the links. Until then the "contract version 1" / "schema in [...]" references are promises with no backing.

- **[P2] `launch.md` and `synthesize.md` have accreted into orchestration god-files mixing the core 3-phase lifecycle with ~15 optional integrations** — `skills/flux-engine/phases/launch.md` (308 lines), `skills/flux-engine/phases/synthesize.md` (623 lines) — The 3-phase lifecycle (triage→dispatch→synthesize) is *conceptually* clean, but Phase 2's `launch.md` now interleaves the core dispatch loop with Composer dispatch (2.0.4), Clavain routing (2.0.5), qmd knowledge (2.1), interspect overlays (2.1d), intertrust multipliers (2.1e), FluxBench challenger shadow dispatch via OpenRouter (2.2-challenger), research-context dispatch (2.2a), and synthetic-refusal detection. `synthesize.md` similarly folds in Lorenzen discourse config, interstat token accounting, beads creation, and silent knowledge compounding. Each is individually marked "OPTIONAL/progressive enhancement," but a reader (human or agent) must wade through every optional branch to follow the mandatory path. This is accidental complexity per the fd-architecture YAGNI lens: the core path is buried. Smallest viable fix: extract the optional integrations into a sibling file (e.g. `phases/integrations.md`) referenced by one line per hook point, leaving `launch.md`/`synthesize.md` to carry only the mandatory dispatch/synthesis spine. The pattern already exists (`expansion.md`, `slicing.md` are pulled out) — apply it consistently to the optional integrations.

- **[P2] Config is split across two sibling namespaces (`config/flux-drive/` vs `config/flux-review/`) with no shared layering contract** — `config/flux-drive/*.yaml` (12 files), `config/flux-review/{defaults,guide}.yaml` — `flux-review-engine` reads `config/flux-review/defaults.yaml` with a clean 3-tier merge (flags > project > plugin defaults, SKILL.md:24-28), while `flux-engine` reads a sprawling `config/flux-drive/` tree (budget, reaction, model-registry, 4 discourse-*, 2 fluxbench-*, agent-roles). The directory is named `flux-drive` but the skill consuming it is `flux-engine`, and the two config roots share no common base or documented relationship — `flux-review`'s routing/quality model is entirely parallel to `flux-drive`'s budget/agent-roles model even though both ultimately route the same agents to the same models. The result is two unrelated config dialects for one engine family. This is tolerable today but is the seam most likely to produce inconsistent behavior (e.g. a model upgraded in `flux-review/defaults.yaml` but not in `flux-drive/agent-roles.yaml`). Smallest fix: at minimum document the relationship (which config governs which skill) in AGENTS.md's config section; longer-term, rename `config/flux-drive/` → `config/flux-engine/` to match its consumer, or hoist shared model-routing into one file both skills read.

- **[P2] The 5 cognitive agents duplicate a verbatim template skeleton that will drift** — `agents/review/fd-{systems,decisions,people,resilience,perception}.md` — All five cognitive agents share byte-identical scaffolding: the "First Step (MANDATORY)" paragraph, the Severity Guidance block (the `P0**: Reserved for missing analysis creating immediate concrete risk (rare)` line appears verbatim in all 5), and the `Limit to 5-8 findings per review` / "Note technical agent crossovers" Focus Rules (confirmed: all 5 match). Most fragile is the **"What NOT to Flag" cross-reference list** — each agent manually enumerates the other four cognitive domains by name (fd-people.md:66 lists "systems dynamics (fd-systems), decisions (fd-decisions), innovation (fd-resilience), perception (fd-perception)"). When a sixth cognitive agent is added or one is renamed, all five files must be hand-edited in lockstep or the anti-overlap routing rots. The 12-item "Key Lenses" content is legitimately distinct per agent and should stay; the *boilerplate* is the duplication worth consolidating. Smallest fix: extract the shared skeleton (severity mapping, focus rules, the cross-reference roster) into `agents/review/references/cognitive-shared.md` and have each agent reference it, leaving only the per-agent Review Approach + Key Lenses inline. A structural test guarding the roster list against drift would be the cheaper interim mitigation.

- **[P2] OUTPUT_DIR convention is forked three ways with no shared resolver, inviting cross-skill collisions** — `skills/flux-engine/SKILL.md:99,108`, `skills/flux-review-engine/SKILL.md` (Step 5 / `track-synthesis.md:83`) — Review writes to `docs/research/flux-drive/{stem}`, research to `docs/research/flux-research/{stem}`, multi-track to `docs/research/flux-review/{slug}/{date}`. The three paths are derived by three independent prose blocks (and a fourth, stale copy in SKILL-compact.md). `track-synthesis.md:7` even hedges that per-track flux-drive runs "may" write to `flux-drive/{stem}` or to per-track subdirs "if the track agents specified output dirs" — i.e. the wrapper is not certain where its own children wrote. Since `flux-review-engine` invokes `flux-engine` per track, and `flux-engine` content-addresses OUTPUT_DIR by `sha256(INPUT_PATH)[:8]` (SKILL.md:114), **all tracks reviewing the same INPUT_PATH resolve to the identical `flux-drive/{stem}-{hash}` directory and race each other** — exactly the concurrent-run hazard the SKILL warns about (line 124) but here triggered internally by the wrapper, not by the user. Smallest fix: have `flux-review-engine` pass an explicit per-track `--output-dir docs/research/flux-review/{slug}/track-{letter}` to each inner `flux-engine` dispatch (the `--output-dir` flag already exists, SKILL.md:67), and make `track-synthesis.md` read from those deterministic paths instead of guessing.

- **[P3] Script directory has outgrown its documented inventory (40 scripts vs ~14 in AGENTS.md), with vestigial entries still shipped** — `scripts/` (40 `.sh`/`.py` files), `AGENTS.md:393` Scripts Reference (~14 rows) — The Scripts Reference table documents roughly 14 scripts; disk has 40, including 7 `fluxbench-*` scripts and undocumented `team_synthesize.py`, `cluster_specs.py`, `benchmark_synthesis.py`, `triage-stats.py`, `discourse-health.sh`, `_decisions_log.py`, `_verification.py`, `sanitize_untrusted.py`, etc. CLAUDE.md/AGENTS.md also still describe `launch-qmd.sh` as "vestigial — moved to interknow" yet it remains in the tree (`scripts/launch-qmd.sh`). A 3x undocumented surface makes it impossible to reason about which scripts are live, which are internal helpers (the `_`-prefixed ones), and which are dead. Smallest fix: regenerate the Scripts Reference from disk (or add a `scripts/README.md` index that the existing one is — but it too is partial), and delete confirmed-vestigial scripts like `launch-qmd.sh` rather than documenting them as dead-but-present.

- **[P3] FluxBench/discourse/challenger subsystems are tightly woven into the core dispatch path despite being experimental** — `config/flux-drive/budget.yaml:90-101` (challenger), `:66-88` (cross_model_dispatch, model_discovery), `config/flux-drive/reaction.yaml:37-43` (4 discourse protocols), `skills/flux-engine/phases/launch.md:188` (Step 2.2-challenger) — A large amount of in-flight experimental machinery (FluxBench challenger shadow dispatch through the openrouter MCP, cross-model tier routing, weekly interrank model discovery, four discourse protocols — sawyer/lorenzen/fixative/topology) is enabled-by-default in config and inlined into the mandatory phase files. `cross_model_dispatch.enabled: true, mode: enforce` and `challenger.enabled: true` mean the hot path *always* evaluates these branches even on a routine review. The fd-architecture concern is premature integration: experimental subsystems should sit behind a single clearly-labeled extension seam, not be threaded through Step 2.2 of the core launch sequence. This compounds finding P2 (god-files). Smallest fix: gate the experimental branches behind one `experiments:` config block and one phase file (`phases/experiments.md`) so the core lifecycle reads cleanly and experiments can be disabled wholesale; default the unproven ones (challenger, model_discovery) to `enabled: false` until qualified.

- **[P3] Command-vs-skill naming collision worked around per-skill instead of via a single convention** — `commands/flux-drive.md:11`, `commands/flux-review.md:11`, `commands/flux-research.md` — Both `flux-drive` and `flux-review` carry near-identical inline notes explaining that the underlying skill was renamed (`flux-engine`, `flux-review-engine`) "to avoid a name collision … the command would shadow the skill at invocation time." This is a real platform constraint handled correctly, but the rationale is copy-pasted into each command and the `-engine` suffix convention is documented only as scattered prose, not as a stated project rule. A future author adding a `/flux-foo` command may not know to name its skill `flux-foo-engine`. Smallest fix: state the "command `X` → skill `X-engine`" convention once in AGENTS.md (Design Decisions / Components) and reduce the per-command notes to a one-line pointer. Minor, but it converts an undocumented tribal rule into a boundary contract.
