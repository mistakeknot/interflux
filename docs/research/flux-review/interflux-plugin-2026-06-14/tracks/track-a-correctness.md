# Track A — Correctness & Concurrency (fd-correctness)

Reviewer: Julik (fd-correctness) — data consistency, state machines, race conditions, async/concurrency, invariants, failure modes.
Date: 2026-06-14 | Target: interflux orchestration engine.

## Correctness assessment

The completion protocol is well thought through. The core invariant — "exactly one terminal `.md` per agent per run, owned by the agent's atomic `.partial → .md` rename" — is sound, and the Retry Race Protocol (`shared-contracts.md:83-111`) correctly defends against a slow original Task clobbering a retry by renaming the source out of the way first (so the original's `mv` fails harmlessly). The OUTPUT_DIR quire-mark (`launch.md:17-24`, validated in `synthesize.md:27-32`) is a genuinely good defense against cross-run contamination on the content-addressed path. The MCP server uses tempfile+rename for state and an O_EXCL lockfile with stale-break, which is the right shape. However, several real correctness defects remain: the engine relies on **filesystem `.md` presence as the sole completion signal but never guards against partial/crashed writes producing a truncated-but-renamed `.md`**; the **OUTPUT_DIR content-address is `sha256(INPUT_PATH)` only**, so two different inputs that normalize to the same `INPUT_STEM`/path string, or two concurrent runs on the same target, share a directory and race; the **spend-ceiling accounting in the OpenRouter MCP has a check-then-act gap that lets concurrent requests overshoot the ceiling unboundedly**; and the **convergence/severity regex `[Pp][0-2]` misclassifies multi-digit severities** (`P10` → `P1`). Below, ordered by severity.

## Findings

- **[P1] Spend-ceiling check-then-act allows unbounded overshoot** — `mcp-servers/openrouter-dispatch/index.ts:176-184, 262-268` — The ceiling is checked and a rate token consumed inside one `withStateLock`, but `cumulativeSpendUsd` is incremented in a *separate* `withStateLock` only *after* the network call returns (`:264-268`). Between the pre-flight check and the post-call increment the lock is released. Failure scenario: with `SPEND_CEILING=$10` and N concurrent `review_with_model` calls, all N pass the `cumulativeSpendUsd >= SPEND_CEILING` check (spend still 0 / under ceiling), all N hit the API, then all N increment. The ceiling is enforced only on the *next* call after settlement, so the actual spend can exceed the ceiling by up to (in-flight concurrency × per-call cost). This is exactly the "ceiling became effectively unbounded" risk the persistence comment at `:27-30` claims to have fixed — persistence fixed cross-instance *visibility*, not the check/act window. The rate limiter does not have this bug (token is decremented atomically with the check). Fix: reserve an estimated cost at admission time (decrement a budget inside the same lock as the check, using `max_tokens`-based estimate), then reconcile to actual `total_cost` after the call; or serialize calls when near the ceiling.

- **[P1] `.md` presence is treated as completion even for a truncated/partial write** — `shared-contracts.md:30-32`, `flux-watch.sh:178-192`, `synthesize.md:5-13` — The binding completion signal is "`.md` exists, `.partial` does not", detected via inotify `close_write`/`moved_to`. The sentinel `<!-- flux-drive:complete -->` is explicitly downgraded to "diagnostic metadata only" and the validator only checks the run-uuid first line + `### Findings Index` presence (`synthesize.md:27-40`). If an agent crashes mid-write *after* renaming `.partial → .md* (or writes directly to `.md` without the partial dance, or the rename happens but the body flush is incomplete on a crash), a structurally-truncated file is accepted as a terminal success and its (missing) findings are silently counted as "no findings / safe". Because the sentinel is the only end-of-content marker and it is deliberately *not* required, there is no integrity gate on the body. Failure scenario: agent writes Findings Index header, machine OOM-kills it before it writes Verdict, but a prior `mv` already landed → synthesis reads a Valid-but-empty file, no error stub, verdict skews `safe`. Fix: make synthesis require *both* a parseable `Verdict:` line and (for non-error files) treat absence of the completion sentinel as Malformed→error-stub rather than a warning, restoring the sentinel to a load-bearing role.

- **[P1] OUTPUT_DIR content-address ignores agent roster and input content; collisions silently share a directory** — `SKILL.md:97-117`, `launch.md:12-15` — `RUN_HASH = sha256(INPUT_PATH or RESEARCH_QUESTION)[:8]` and `OUTPUT_DIR = .../flux-drive/{INPUT_STEM}-{RUN_HASH}`. Two distinct concerns: (1) For `INPUT_TYPE=text`, the hash is over the *text string*, but `INPUT_STEM` is "topic as kebab-case (max 50 chars)" derived separately — two different long questions sharing a 50-char kebab prefix get different hashes (fine), but two *file* inputs whose paths differ only beyond a truncated stem produce different dirs only because the hash is on the full path (fine) — the real hazard is (2): the spec itself documents (`SKILL.md:124`) that two concurrent invocations on the *same* target deterministically share OUTPUT_DIR and "can race", with the only mitigation being a manual `--output-dir`. The pre-dispatch clean (`launch.md:13-15` `find ... -delete`) of run B will delete run A's in-flight `.md`/`.partial`/`peer-findings.jsonl` files. The quire-mark catches stale *prior-run* files at synthesis, but it does NOT prevent run B's clean from destroying run A's live outputs mid-flight, nor two live runs interleaving renames into the same names. Failure scenario: user runs `/flux-drive .` twice in two panes; second run's Step 2.0 `find -delete` wipes the first run's completed agent outputs; first run's synthesis sees missing files. Fix: incorporate the run UUID into OUTPUT_DIR (or refuse to clean if any `.partial` newer than N seconds exists / a lock file is held), and make `--output-dir` collision-safe by default rather than opt-in.

- **[P1] Convergence/severity extraction matches a prefix of multi-digit severities (`P10`→`P1`)** — `scripts/findings-helper.sh:119` — `if (match(line, /[Pp][0-2]/))` extracts the severity by grabbing the first `P0`/`P1`/`P2` substring. On a malformed or future `P10`/`P12` line it matches the `P1` inside `P10` and counts it as P1; on `P3+` it correctly skips. More subtly, the regex is unanchored, so `P0` appearing anywhere in the Title (e.g. a finding titled `"P0 budget overrun handling"`) is read as the severity even if the real index token was `P2`. Because convergence drives the expansion decision (`expansion.md:160-191`) and the synthesis overlap ratio, a misread severity can inflate `overlapping`/`total_findings` and trigger spurious Stage-2 expansion or skew the verdict. Fix: anchor to the index grammar — match `^-[[:space:]]*([Pp][0-9]+)[[:space:]]*\|` and capture the full digit run, then range-check.

- **[P2] `report_existing` excludes `.partial` by basename suffix but the glob also yields no protection against `.md.partial.aborted-*`** — `flux-watch.sh:54-67` — In the no-inotify/initial-scan path, the loop globs `"$OUTPUT_DIR"/*.md` and skips `*.partial` (`:60`). But `*.md.partial.aborted-<ts>` does NOT end in `.md` so it is excluded by the glob (good), yet a file literally named `<agent>.md` produced by a *foreign* concurrent run (no quire-mark) is counted toward `seen` here with no run-uuid check — flux-watch has no knowledge of the quire-mark. So in a shared-OUTPUT_DIR race (see P1 above), flux-watch can satisfy its expected count from another run's files and exit success while this run's agents are still writing. The quire-mark check only happens later in synthesis, by which point flux-watch already returned "all complete". Fix: flux-watch should accept an optional expected-agent-name allowlist (it already takes `EXPECTED_AGENTS` for stall rescue) and only count names in that set, rather than any `*.md`.

- **[P2] Stall-rescue can double-count or under-count under the bounded-read timeout path** — `flux-watch.sh:154-193` — When `STALL_RESCUE=1`, the read uses `-t STALL_TIMEOUT`. On timeout it calls `rescue_stalled` (which writes a stub and increments `seen` via `write_stall_stub`, pre-marking `reported[]`). But `write_stall_stub` itself does `cat > "$stub"` which triggers a `close_write` event that the loop will later read; the guard is `reported[$base]=1` set *before* the write (`:79-80`), so the later event is filtered (`:183`). This is correct only if the agent never *also* completes between the stall write and the event drain. Race: agent finishes its real `.md` (atomic rename) microseconds after `rescue_stalled` checked `[[ -e .../${agent}.md ]]` (`:123`) but before `write_stall_stub`'s `[[ -e "$stub" ]]` guard (`:76`) — the guard catches it and returns 0 *without* incrementing seen, but `reported[$base]` was never set, so the subsequent real-file event increments seen normally. Net: usually fine, but if both the stub guard returns early AND the real file's event was already consumed and filtered earlier, the agent is never counted and flux-watch can hang to TIMEOUT. The interleaving is narrow but real. Fix: have `write_stall_stub` set `reported[]` only after it actually creates the stub, and reconcile `seen` against a final directory scan before declaring timeout (the code does call `report_existing` at `:197`/`:227`, which partially mitigates).

- **[P2] Retry Race Protocol leaves the original Task running with no abort enforcement** — `shared-contracts.md:83-108`, `launch.md:270` — The protocol `touch`es `{agent}.abort` and renames the partial aside, relying on the original Task's eventual `mv .partial → .md` failing because the source is gone. But nothing *stops* the original agent: if the original agent is between Write-of-`.partial` and the rename when Step 2.3 fires, it may (a) re-create a fresh `.partial` and rename it to `.md` *after* the retry already produced `.md`, violating the "at most one terminal .md" invariant, since `mv` does not refuse to overwrite without `-n` (the doc itself notes this at `:76`). The `.abort` marker is written but there is no documented contract that agents poll it. Failure scenario: original agent was slow on its Read step, finishes after retry, writes its own `.partial`→`.md`, clobbering the retry's good output with stale findings. Fix: specify that the rename MUST use `mv -n` (no-clobber) for the agent-side rename, or that synthesis prefers the file by mtime/quire-mark and the orchestrator verifies the retry's content survived.

- **[P2] `cmd_record` reads line count and increments without locking the agent file; concurrent reviews lose updates** — `scripts/flux-agent.py:615-663`, atomic write at `:202-212` — `cmd_record` does read-frontmatter → compute `use_count+1` → `_update_frontmatter` (which does tempfile+`os.rename`). The write is atomic, but the read-modify-write is not: two flux-drive runs finishing near-simultaneously (Phase 3 each calls `flux-agent.py ... record`) both read `use_count=5`, both write `6`, losing one increment. Same lost-update on `tier` promotion. Impact is low (best-effort registry, idempotent-ish), but tier auto-promotion (`:644-647`) can stick one review short of `proven`. Fix: flock the agent file (or the agents dir) around read-modify-write, consistent with the flock discipline already used in `findings-helper.sh:39-42`.

- **[P3] `token-count.py` fallback is unreachable for the all-malformed-JSONL case** — `scripts/token-count.py:44-93` — `sum_usage` was deliberately changed to per-line try/except so one bad line doesn't nuke the file (good), but the consequence is that a JSONL that is *entirely* malformed (e.g. a truncated/binary file) no longer raises — it returns all-zeros with `_valid_lines=0`. `main` (`:85-93`) only falls back to chars/4 on an *exception*, which can no longer occur for parse failures; so a fully-corrupt transcript yields `total:0` instead of the intended chars/4 estimate, and exits 0 (success). The docstring at `:46-47` says "caller can detect via total == 0", but `synthesize.md:328-333` does not — it writes `total_tokens=0` into interstat. Fix: in `main`, if `result.get("_valid_lines",0)==0`, route to `fallback(fallback_file)` and exit 1 as originally intended.

- **[P3] Temp-file cleanup glob can delete a concurrent run's files** — `synthesize.md:509-516`, `launch.md:81,92` — Temp review files are `/tmp/flux-drive-${INPUT_STEM}-${TS}*.md/.diff` and Step 3.7 cleans `/tmp/flux-drive-${INPUT_STEM}-*` (wildcard over TS). Two runs on inputs sharing the same `INPUT_STEM` (the stem is a 50-char-truncated basename/kebab, so collisions are plausible across sibling files like `plan-v1.md` and `plan-v2.md` if truncated, or identical basenames in different dirs) will have run A's Step 3.7 delete run B's still-in-use temp files while B's agents are mid-Read. Failure scenario: agent in run B gets a "file not found" on its first Read and produces an error stub. Fix: include `RUN_HASH`/`FLUX_RUN_UUID` in the temp-file name and scope the cleanup glob to it.

- **[P3] Empty-input / all-skip triage has no explicit floor enforcement at the dispatch boundary** — `SKILL.md:247-252`, `references/budget.md` (min-2 rule) — The scoring section says "base >= 3 always included … >= 1 only for thin sections" and budget enforces "min 2 agents", but the pre-filter (`SKILL.md:210-225`) can legitimately drop *every* technical agent for a trivial diff (no DB/security/perf/game signals, `fd-architecture`/`fd-quality` "always pass" — so two survive). For a `text` input that classifies as none of the document types, cognitive agents are skipped and only `fd-architecture`/`fd-quality` remain; if those are also excluded by a routing-override (`SKILL.md:194-208`), the candidate pool can reach zero with only a one-line warning, and there is no documented "abort with message" vs "force min-2 from a default set" tiebreak. Score ties at the slot ceiling are resolved only implicitly (no documented stable sort at *selection* time, unlike the explicit pre-dispatch sort at `expansion.md:196-204`). Fix: document an explicit "if candidate pool is empty after overrides, fall back to {fd-architecture, fd-correctness} and warn" floor, and add a stable tiebreaker (name asc) to Stage-1 selection to make triage fully deterministic as the docs claim (`SKILL.md:284`).

## Notes / non-issues verified

- The OUTPUT_DIR quire-mark + synthesis Foreign classification (`synthesize.md:27-32`) correctly defends synthesis against *prior-run* stale files — the gap is only live concurrent runs (P1/P2 above).
- The OpenRouter lockfile uses `wx` (O_EXCL) with a 60s stale-break and best-effort release (`index.ts:50-89`) — correct shape; the bug is the spend check/act window, not the lock.
- `token-count.py` per-line resilience (`:31-43`) is a genuine improvement over whole-file fallback; the only gap is the all-malformed edge (P3).
- inotify PID resolution via `pgrep -P $$` (`flux-watch.sh:141-142`) correctly fixes the process-substitution PID problem that would have orphaned watchers.
