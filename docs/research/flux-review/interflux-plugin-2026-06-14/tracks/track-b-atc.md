source_domain: Air-traffic control / terminal-area approach sequencing
distance_rationale: Same abstraction level as interflux's staged dispatch (sequencing many concurrent agents through limited capacity) but under hard, safety-critical operational constraints.
expected_isomorphisms: priority sequencing under capacity limits, separation/contention control, holding patterns when capacity is exceeded, go-arounds (retries), controller workload saturation (orchestrator context limits), handoff protocols, minimum-safe-altitude floors.

---

An approach controller sequences more arrivals than the runway can absorb. The governing disciplines are: never let an aircraft fall out of positive control (every aircraft is always sequenced, held, or handed off — never "lost"); sequence by merit and fuel state, not arrival order; protect a minimum-separation floor that no traffic level can erode; and never saturate the controller's own scan. Below I map these against interflux's dispatch architecture.

- **[P1] Track×agent fan-out has a holding pattern but no queue discipline — late agents can be silently starved** — `skills/flux-review-engine/phases/track-dispatch.md:189-196` and `skills/flux-engine/phases/launch.md:166-171` — ATC principle: when the runway is saturated you put aircraft in a *named, ordered* holding stack (EFC times, fuel-priority order) so every aircraft has a guaranteed exit, not a best-effort one. interflux gap: the inner cap (`MAX_CONCURRENT_AGENTS`, default 6) and outer cap (`min(TRACK_COUNT,4) × PER_TRACK_AGENT_CAP`, default 12) are enforced as "before issuing the next dispatch, if in_flight >= cap, wait for any in-flight task to complete" (launch.md:168-169). That is a *throttle*, but there is no explicit queue ordering for the *waiting* set — when in-flight slots free up, the spec never says which pending agent claims the slot. The merit-order sort at `expansion.md:198-204` applies only to the Stage 2 candidate pool, not to the throttle's release order. With repeated waits, a low-listed agent could be perpetually deferred behind newly-eligible higher-priority ones. Concrete fix: make the throttle FIFO-by-merit-score explicit — "when a slot frees, dispatch the highest-merit *not-yet-dispatched* agent, and never re-rank an already-queued agent below a newly-arrived one" (an EFC-style monotonic guarantee). State this in launch.md § Concurrency cap as an invariant.

- **[P1] No go-around policy for the multiplicative track×agent retry storm** — `skills/flux-engine/phases/launch.md:262-307` and `skills/flux-review-engine/phases/track-dispatch.md:189-196` — ATC principle: a go-around is a *bounded, declared* maneuver — the aircraft climbs to a published missed-approach altitude and re-enters the sequence at a known point; you never allow unbounded re-attempts that re-congest final approach. interflux gap: the inner flux-drive retry policy is well-specified (one sync retry on `.partial`-only, one tier-downgrade retry on synthetic refusal, launch.md:270/288). But at the *track* layer, `track-dispatch.md` only says "If any track fails, the command degrades gracefully to surviving tracks" (SKILL.md:153) — there is no retry at all for a wholly-failed track, and no coordination preventing all 4 tracks from entering their inner retry loops simultaneously. Four tracks each retrying ~3 agents synchronously is a 12-agent retry burst landing on the same rate-limited API key with no outer throttle (the outer cap governs initial dispatch, not retries). Concrete fix: define a track-level go-around — a failed track gets exactly one re-dispatch, gated behind the same `MAX_CONCURRENT_TRACKS` semaphore, and inner retries must also count against the outer cap so a retry storm cannot exceed the 12-agent ceiling.

- **[P1] Orchestrator scan saturation is mitigated for synthesis but NOT for dispatch/monitoring** — `skills/flux-engine/phases/synthesize.md:46-47,103` vs `skills/flux-engine/phases/launch.md:260,266` and `expansion.md:305` — ATC principle: the single hardest cap in terminal control is the *controller's own scan rate* — TMUs cap the number of aircraft a sector may work precisely because a saturated controller is the failure mode, not the runway. interflux correctly offloads the read-heavy phase: synthesis is delegated to a subagent and "the host agent never touched any individual agent output file" (synthesize.md:103), and verdict extraction is `tail -7` (shared-contracts.md:139). But the *dispatch/monitor* phase keeps the orchestrator in the loop for every completion: it must "maintain a mapping of agent_name → agent_id for all dispatched agents" (launch.md:260), display per-completion progress lines (launch.md:266), run incremental-expansion scoring as each Stage 1 agent completes (expansion.md:96-106), and read each completed agent's Findings Index for escalation advisories (expansion.md:336-342). At the 16-agent / 4-track tier this is real context accretion in the controlling agent during the highest-traffic phase. interflux gap: there is no stated ceiling on orchestrator working-context during dispatch, analogous to the synthesis offload. Concrete fix: cap the orchestrator's in-context completion tracking (e.g., persist the agent_name→agent_id map to a temp file rather than holding it inline) and have incremental-expansion scoring read only the Findings Index head, never prose — make explicit that the controller reads "strips," not full files, during the traffic peak too.

- **[P2] Stage-1 protection is a safety floor, but the floor is only 2 agents and is not separation-aware** — `config/flux-drive/budget.yaml:29,46-48` and `skills/flux-engine/SKILL.md:24,272` — ATC principle: the minimum-safe configuration is not just "land *something*" — it's "maintain required separation between the things you do land." A capacity collapse should degrade to a *known-safe* minimum service, not an arbitrary one. interflux gap: the safety floor is `min_agents: 2` (budget.yaml:29) plus `exempt_agents: [fd-safety, fd-correctness]` never deferred (budget.yaml:46-48, SKILL.md:24). That is a sensible floor *for code inputs*, but the two exempt agents are both pre-filtered *out* for non-security, non-DB document inputs (SKILL.md:218-220 skips fd-safety unless security/credentials/deploy/trust; fd-correctness unless DB/migrations/concurrency). So for a plain plan/PRD under budget collapse, the "safety floor" is whichever 2 agents happen to score highest — there is no guaranteed cross-cutting reviewer. Concrete fix: make the floor *role-aware*: always retain at least one domain-general cross-cutting agent (fd-architecture or fd-quality, both "always pass" per SKILL.md:219) in the min-2 floor, so the minimum-viable review always covers the structural spine, not two narrow lenses that happen to win triage.

- **[P2] No fuel-state / wall-clock priority — long-running agents can't preempt the queue** — `skills/flux-engine/phases/expansion.md:196-204` and `skills/flux-engine/phases/launch.md:264` — ATC principle: a low-fuel aircraft jumps the sequence regardless of arrival order — *time-criticality* is a first-class priority dimension orthogonal to merit. interflux gap: dispatch merit-order (expansion.md:198-204) ranks by expansion_score → role_priority → name. There is no notion of an agent's *deadline pressure* — timeouts are fixed per type (Task=300s, Codex=600s, launch.md:264) and an agent approaching its timeout has no way to be reprioritized or given a slot ahead of a fresh low-merit agent. The Oracle (cross-AI, 600s + Chrome) is the "heavy widebody" here and is the most likely to time out under contention, yet nothing sequences it preferentially or reserves separation around it. Concrete fix: add a coarse time-criticality tiebreaker to the merit sort — agents with the longest timeout budgets (Oracle, deep reviews) dispatch *first* into the concurrency window so their long pole overlaps maximal headroom, mirroring "land the low-fuel/long-final traffic early."

- **[P2] Handoff between phases lacks a positive-control checklist — completion is inferred by file presence, not acknowledged** — `skills/flux-engine/phases/shared-contracts.md:30-32,78` and `synthesize.md:5-13` — ATC principle: a handoff is not complete until the receiving controller *verbally accepts* the aircraft ("radar contact"); silent transfer is a control loss. interflux's Phase 2→3 handoff is sound in design (the `.partial → .md` rename is "the binding completion signal," shared-contracts.md:30) but the *receiver* (synthesis) only checks file *count* — "Confirm N files... If count < N, Phase 2 did not complete properly" (synthesize.md:11-13). There is no positive handoff record asserting "the orchestrator has reconciled all N agents and is transferring control of these specific N artifacts." interflux gap: a stall stub, an aborted partial, and a genuine completion can all yield "N files," masking which agents were truly worked vs rescued (the stall-rescue path at launch.md:268 writes a real `.md`). Concrete fix: emit an explicit handoff manifest at end of Phase 2 (list of {agent, terminal_state ∈ completed|stalled|refused|errored, run_uuid}) that Phase 3 must validate against, so the receiving phase positively confirms each aircraft's disposition rather than counting blips on the scope.

- **[P3] No spacing/anti-collision guard on the shared OUTPUT_DIR for concurrent runs — flagged but not enforced** — `skills/flux-engine/SKILL.md:124-126` and `phases/launch.md:12-15` — ATC principle: two aircraft cleared to the same runway is the cardinal sin; the system *physically prevents* it (one runway = one occupancy clearance), it doesn't merely warn. interflux gap: the content-addressed OUTPUT_DIR means two concurrent flux-drive runs on the same target share a directory and "can race" (SKILL.md:124); the mitigation is a documented caveat ("pass `--output-dir <unique>`") plus a pre-dispatch `find ... -delete` (launch.md:14) — i.e., a NOTAM, not a lock. The run-uuid quire-mark (launch.md:17-24) lets synthesis *detect* contamination after the fact, but nothing prevents run B's pre-clean from deleting run A's in-flight `.partial` files. Concrete fix: take a real occupancy lock — `mkdir {OUTPUT_DIR}/.lock` (atomic) at dispatch start, refuse/auto-suffix if held, release at synthesis end. Convert the "can race" caveat into an enforced single-occupancy clearance.

- **[P3] Stall rescue is opt-in — the safety net defaults off** — `skills/flux-engine/phases/launch.md:268` and `shared-contracts.md:81` — ATC principle: the conflict-alert / minimum-safe-altitude warning is *always armed*; you do not run terminal ops with the safety net disabled to save a little processing. interflux gap: stall rescue (write a stall stub + `kind:stall` peer-finding when an agent produces neither `.partial` nor `.md` within `STALL_TIMEOUT`) is "Off by default for back-compat; turn on for any review where partial-progress synthesis is preferable to all-or-nothing" (launch.md:268). The default behavior is therefore: a silently stalled agent costs the *full* 300s timeout × the agent before synthesis sees it as missing — exactly the all-or-nothing failure mode the rescue exists to prevent. Concrete fix: default `STALL_RESCUE=1` (it degrades gracefully — a stall stub is strictly more information than silence) and let runs that genuinely want all-or-nothing opt *out*. Arm the safety net by default.
