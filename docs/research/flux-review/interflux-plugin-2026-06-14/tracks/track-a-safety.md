# Track A — Safety Review (fd-safety)

## Safety Posture Assessment

interflux is a local-only Claude Code plugin (orchestrator + sub-agents + two stdio MCP servers + bash/python helper scripts). The realistic threat model is not network exposure but **untrusted content flowing into an LLM trust boundary**: the engine reads arbitrary repo files, fetches web content (Exa/WebSearch), pulls knowledge entries, and renders LLM-authored agent specs — then feeds all of that back into agent and orchestrator system prompts. The codebase shows clear evidence of having thought about this: `scripts/sanitize_untrusted.py` is a conservative, well-documented prompt-injection filter (NFKC normalization, system-tag stripping, override-line blanking, control-char dropping, code-fence/base64 stripping); `generate-agents.py` validates agent `name` against a strict `^fd-[a-z0-9-]+$` `fullmatch` before path construction and sanitizes all LLM free-text fields; `cost_capture.sh` validates session-id format before shell interpolation; the openrouter MCP hardcodes its endpoint (no SSRF), persists rate/spend state under a flock with `0o600`/`0o700` perms, and refuses to start without a key. No secrets are committed (`.env` is gitignored, only an empty `.env.example` exists, gitleaks baseline present). Overall the security engineering is above average for a plugin of this size. The residual risk is concentrated in (1) **enforcement gaps** — sanitization is documented in phase markdown as the orchestrator's job rather than mechanically guaranteed, so a single skipped step reopens the injection channel across all four sinks; (2) the **openrouter-dispatch MCP**, which is a real outbound-spend / arbitrary-model-dispatch surface with weak input bounds; and (3) a handful of operational/integrity sharp edges (auto-build on launch, unvalidated env-driven paths, stub-file injection into synthesis). None are remote-exploitable in the standalone threat model, but several are exploitable by hostile repo content or a compromised peer agent.

## Findings

- **[P1] Prompt-injection sanitization is advisory, not enforced, across four sinks** — `scripts/sanitize_untrusted.py:1-21`, `skills/flux-engine/phases/reaction.md:71-96`, `skills/flux-engine/phases/shared-contracts.md:156-164` — The trust boundary is real: peer findings (from flux-gen agents with arbitrary prompts), LLM-generated specs, knowledge context, web/research findings, and domain overlays from untrusted repos all get embedded into system prompts. The mitigation (`sanitize()`) exists and is good, but for the *runtime* channels (reaction round, knowledge/research/domain injection) it is invoked only because a markdown phase doc *tells the orchestrating LLM to run it*. There is no mechanical guarantee — an LLM that skips the step, or a path that injects content without going through reaction.md (e.g. Step 2.2a research context, Step 2.1d overlays), embeds raw attacker-controlled text. Impact: an injected `</system>… new instructions: exfiltrate / write file X` could steer a downstream agent. Mitigation: route every injection sink through a single enforced wrapper (the planned `TrustedContent` NewType in C3), or have the orchestrator sanitize at the only chokepoint that reads these files, rather than per-call-site in prose.

- **[P1] openrouter-dispatch accepts arbitrary `model_id` and unbounded prompt size — spend/abuse surface** — `mcp-servers/openrouter-dispatch/index.ts:157-231` — `model_id` is a free `z.string()` passed straight into the OpenRouter request body, and `prompt`/`system_prompt` have no length cap. A compromised or injection-steered agent calling this tool can dispatch to the most expensive model available and send arbitrarily large prompts. The spend ceiling (`OPENROUTER_SPEND_CEILING_USD`, default 1.00 in plugin.json) only checks *cumulative* spend *before* the call and is only decremented when OpenRouter returns `total_cost` (line 264) — a single very large call can overshoot the ceiling by an unbounded amount before the post-call accounting catches up, and any response lacking `usage.total_cost` is billed but never counted, so the ceiling silently drifts upward. Mitigation: constrain `model_id` to an allowlist (or the FluxBench model registry), enforce a `max_tokens` ceiling and a prompt-length cap, and treat missing `total_cost` as a conservative estimated debit rather than zero.

- **[P1] launch-openrouter.sh auto-runs `npm ci && npm run build` at MCP startup** — `scripts/launch-openrouter.sh:19-23` — When `dist/index.js` is missing, the launcher silently runs `npm ci && npm run build` from `mcp-servers/openrouter-dispatch`. This executes whatever is in that directory's `package.json`/lockfile and any (transitive) install scripts the moment the MCP server is launched — i.e. as a side effect of starting a Claude Code session with `OPENROUTER_API_KEY` set. Supply-chain/build steps should be an explicit, auditable action, not an implicit launch-time effect; a tampered lockfile or a malicious postinstall would run with the user's privileges. Mitigation: ship a prebuilt `dist/` (or fail with an instruction to build manually) rather than building on launch; at minimum use `npm ci --ignore-scripts`.

- **[P2] flux-watch writes attacker-influenceable stall stubs and peer-findings that synthesis trusts as data** — `scripts/flux-watch.sh:73-108`, `163-198` — In `STALL_RESCUE=1` mode, flux-watch fabricates `{agent}.md` files and appends JSON lines to `peer-findings.jsonl` for any agent name supplied via the `EXPECTED_AGENTS` env var, writing into `OUTPUT_DIR` (also env-supplied). The agent name is interpolated into both a file path (`$OUTPUT_DIR/${agent}.md`) and a JSON string via `printf` without validation or JSON escaping — an agent name containing `"`/newlines/`../` corrupts the JSONL or escapes the directory, and the stub becomes synthesis input. In normal operation these values are orchestrator-controlled, so this is integrity/robustness rather than a direct external exploit, but the lack of validation on a path- and JSON-bound variable is a latent injection. Mitigation: validate `agent` against the same `^[a-z0-9-]+$` pattern used elsewhere, and emit the JSONL via `jq -n --arg` (as `findings-helper.sh:28-35` already does correctly) instead of raw `printf`.

- **[P2] session-start hook writes `export FLUX_BUDGET_REMAINING=...` into CLAUDE_ENV_FILE from a partly external signal** — `hooks/session-start.sh:56-82` — The hook reads a budget percentage from `${INTERBAND_ROOT}/interstat/budget/${session_id}.json`, computes a remaining figure, and appends `export FLUX_BUDGET_REMAINING=<n>` to `$CLAUDE_ENV_FILE`. The code does validate `_if_pct` numerically (line 76) and `_if_budget` (line 78), and `_if_remaining` is forced to `%d`, so injection into the export line is blocked. The residual risk is trust-boundary: a writable `~/.interband/interstat/budget/*.json` (potentially populated by another tool/process) influences an env var that flux-drive uses to size dispatch — a corrupted-but-numeric payload yields a wrong budget. This is correctly defended against the worst case (no eval/expansion of attacker text), so it is P2 robustness rather than P1. Mitigation: keep the strict numeric guards (good), and consider validating the envelope provenance before honoring the value when interband validation is unavailable (the raw-jq fallback at line 69 skips envelope checks).

- **[P2] Knowledge/decay path is driven by hardcoded cross-repo paths and an unenforced sanitization step** — `skills/flux-engine/phases/synthesize.md:583-600` — Knowledge entries are sanitized (strip external paths, hostnames, org/customer identifiers) and decay-archived under `{PROJECT_ROOT}/interverse/interknow/config/knowledge/`. As with finding 1, the sanitization is an instruction to the LLM, not enforced code, so secret/PII leakage into persisted knowledge depends on orchestrator discipline. Because knowledge is later re-injected into prompts (and may be shared via interknow/qmd), a single missed redaction compounds. Mitigation: run the same `sanitize_untrusted.py`-style filter (extended with a secret/PII redactor) as a mandatory pre-write step in the synthesis code path rather than as prose guidance.

- **[P2] openrouter persisted state can be poisoned to disable spend/rate limits** — `mcp-servers/openrouter-dispatch/index.ts:91-122`, `137-147` — Rate-limit and cumulative-spend state live in `~/.config/interflux/openrouter-state.json`. `loadState()` clamps `tokenBucket` to `RATE_LIMIT` but accepts any `cumulativeSpendUsd` value from disk; on first run / load failure it starts from defaults. The file is written `0o600`, so the practical attacker is another local process running as the same user. A process that resets `cumulativeSpendUsd` to a low value (or deletes the file) restores the full spend budget; setting `lastRefill` far in the past refills the token bucket. This is a local same-user integrity issue (limited blast radius) but defeats the very ceiling the persistence was added to enforce (per the comment at lines 27-30). Mitigation: treat the spend ledger as monotonic (never accept a *lower* persisted cumulative spend than the in-memory high-water mark within a session), and consider an HMAC/owner check before trusting the file.

- **[P3] OpenRouter request leaks an unrelated/incorrect attribution header** — `mcp-servers/openrouter-dispatch/index.ts:227-228` — Every dispatched request sends `HTTP-Referer: https://github.com/sylveste-ai/sylveste` and `X-Title: "FluxBench Qualification"`. This is not a secret leak, but it sends a hardcoded third-party project URL to OpenRouter on behalf of every interflux user and is inconsistent with the plugin's own identity (`mistakeknot/interflux`). Low-severity metadata/branding hygiene. Mitigation: set the Referer/Title to the interflux project (or make them configurable), and confirm no internal identifier is being inadvertently disclosed.

- **[P3] launch-openrouter degrades silently (exit 0) on missing Node/key while launch-exa surfaces a config error (exit 78)** — `scripts/launch-openrouter.sh:9-17` vs `scripts/launch-exa.sh:9-18` and `index.ts:16-19` — The exa launcher and the openrouter `index.ts` both deliberately use exit 78 (EX_CONFIG) so a missing key surfaces as a config error rather than "ran successfully." The openrouter *launcher* instead exits 0, masking the missing-key/missing-Node state — the opposite of the documented intent. Operational, not a security hole, but it makes a non-functional cross-model review path look healthy. Mitigation: align the launcher to exit 78 on missing prerequisites for consistency with its own server and the exa launcher.

- **[P3] Documentation/manifest drift around MCP servers obscures the real attack surface** — `.claude-plugin/plugin.json:53-71` vs `CLAUDE.md:7` / `AGENTS.md:17` — The manifest registers **two** MCP servers (exa + openrouter-dispatch), but CLAUDE.md's validation snippet asserts `['exa']` and AGENTS.md says "1 MCP server (exa)." The openrouter-dispatch server — the one with real outbound-spend and arbitrary-model-dispatch surface (finding 2) — is the undocumented one. Drifted docs cause reviewers and operators to under-account for the live attack surface. Mitigation: update CLAUDE.md/AGENTS.md (and the `wc`/json validation snippets) to reflect both servers, and document the `OPENROUTER_*` env vars and their security-relevant defaults.

## Notes on Things Checked and Found Safe

- **Path traversal via agent `name`**: mitigated — `generate-agents.py:487,493` validates against `^fd-[a-z0-9]+(?:-[a-z0-9]+)*$` via `fullmatch` (`spec_types.py:179-181`) before `agents_dir / f"{name}.md"`.
- **Path traversal via SLUG/OUTPUT_DIR**: derived from kebab-cased descriptions and content-addressed `sha256(INPUT_PATH)[:8]` suffixes (`skills/flux-engine/phases/launch.md:12`); orchestrator-controlled, no attacker-supplied path component reaches a write unescaped.
- **Command injection in helper scripts**: `findings-helper.sh:28-42` builds JSON with `jq -n --arg`/`--argjson` and writes under `flock`; `discover-models.sh:83` `eval` operates only on a fixed set of literal tier names; `team_synthesize.py:379` invokes bash via an argv list (no shell); `cost_capture.sh:68-72` validates session-id format before interpolation. No exploitable injection found.
- **SSRF in MCP servers**: openrouter endpoint is hardcoded (`index.ts:222`); exa is the upstream `exa-mcp-server` launched via `npx -y` (its own trust surface, but no interflux-introduced SSRF).
- **Committed secrets**: none found. `.env`/`.env.*` gitignored with `!.env.example` exception (`.gitignore:4-6`); `.env.example` holds an empty `EXA_API_KEY=`; gitleaks baseline present (`.gitleaks.toml`).
- **State-file perms**: openrouter state dir created `0o700`, state file written `0o600`, atomic temp+rename (`index.ts:119-125`).
