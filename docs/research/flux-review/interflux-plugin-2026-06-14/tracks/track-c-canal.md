# Track C (Distant) — Canal Lock-Staircase Keeping / Hydraulic Flow Management

source_domain: Pre-industrial civil-hydraulic craft — flights of staircase locks (e.g. Neptune's Staircase), pound locks, weirs, pounds, the lock-keeper's batching schedule, finite summit-pound water budget.
distant_rationale: Moving vessels through staged elevation changes under a finite, shared water budget is a centuries-refined craft of staged gating and flow conservation — utterly removed from software, yet structurally a staged-dispatch-under-shared-resource problem.
expected_isomorphisms: lock = dispatch stage; "never both gates open" = phase separation invariant; finite summit-pound water = the single scarcest shared resource (API rate limit) gating throughput regardless of per-lock size; fill/empty time = per-stage latency; two boats meeting in a single-width pound = OUTPUT_DIR / spec-file collision deadlock; keeper batching boats per cycle = concurrency batching.

---

**[P0] The summit pound, not the per-lock gates, sets the whole staircase's throughput** — `skills/flux-review-engine/phases/track-dispatch.md:188-196`, `skills/flux-engine/phases/launch.md:156-172`

The named mechanism: in a lock flight, every lock can be sized generously, but the *summit pound* — the topmost reservoir feeding the whole descent — holds a finite water budget. Each boat passage drains a fixed lockful from the summit; once the summit is low, no per-lock capacity matters, because there is no water to fill the next chamber. The scarcest *shared* reservoir gates the entire staircase, regardless of each lock's individual dimensions.

The structural parallel: interflux's true summit pound is the **single shared Anthropic API rate limit / concurrent-session budget**, drawn down by every in-flight subagent across all tracks. track-dispatch.md:189 explicitly names it ("The Anthropic API rate limit is shared across all in-flight subagents") and computes `min(TRACK_COUNT, MAX_CONCURRENT_TRACKS) × PER_TRACK_AGENT_CAP` = 4×3 = 12. But the inner flux-drive (`launch.md:161`) *also* has its own `MAX_CONCURRENT_AGENTS=6`, plus reaction rounds, Stage-2 expansion, speculative incremental expansion (`budget.yaml:62-64`, up to 2 extra), research escalation (each agent may spawn 1 research sub-agent, `launch.md:146`), and the FluxBench challenger shadow dispatch (`launch.md:188-227`). These draws are *additive against one reservoir* but governed by *independent per-lock caps*. No component computes total summit drawdown.

The insight/risk: the per-track and per-run caps are "lock dimensions" — they bound each chamber but not the reservoir. The actual concurrent-agent peak is `tracks × per_track_cap + Σ(speculative expansion) + Σ(research escalations) + challenger shadows`, which can substantially exceed the nominal 12 the spec reasons about. Recommend a single **summit-budget governor**: a run-global semaphore (one FLUX_RUN_UUID-scoped counter) that *every* dispatch path — track, stage, expansion, research-escalation, challenger — decrements before launch. Without it, raising any one cap for throughput can silently empty the summit and trigger the very rate-limit retry-waste the spec warns of at launch.md:158.

**[P0] "Never open both gates at once" — the phase-separation invariant is documented but not enforced across the speculative/expansion paths** — `skills/flux-engine/phases/launch.md:62-228`, `phases/expansion.md`

The named mechanism: a pound lock's cardinal safety rule is *you may never open the upper and lower gates simultaneously* — doing so dumps the entire upper pound through the chamber, draining the summit and stranding every boat above. The lock-keeper's whole discipline is sequencing: fill, equalize, open *one* gate, close it, then and only then operate the other.

The structural parallel: interflux's phase lifecycle (triage → launch → synthesize) is the gate sequence, and the binding invariant is the `.md.partial → .md` atomic rename (`shared-contracts.md:26-32`): synthesis (the "lower gate") must not read a chamber that is still filling (an agent still "writing"). The contracts are careful here — completion is owned by the agent's rename, not a sentinel (`shared-contracts.md:30-32`), and "at most one terminal `.md` per agent per run" (`shared-contracts.md:76`). That is correct gate discipline. **But** `launch.md:62-64` (Step 2.0.4 Composer) says when a Composer plan is authoritative it *"skip[s] Steps 2.0.5–2.1e ... and skip[s] to Step 2.3"* — bypassing the trust multiplier, slicing, and overlay steps. And incremental/speculative expansion (`budget.yaml:60-64`) launches Stage 2 agents *during* Stage 1, i.e. opens the next chamber's gate while the current one is still equalizing.

The insight/risk: speculative expansion is a deliberate "both gates ajar" optimization — it trades the strict phase boundary for latency. The risk is the same as in a lock: if a speculative Stage-2 agent writes `peer-findings.jsonl` (shared-contracts; broadcast path) *before* Stage 1's findings settle, the peer-finding pool a Stage-1 agent reads is non-deterministic — the equivalent of a boat entering a chamber whose level hasn't equalized. Recommend that any cross-stage shared mutable state (peer-findings.jsonl) carry a stage-tag so synthesis can detect a peer-finding that was written "while both gates were open," analogous to the quire-mark (`launch.md:17-24`) but for stage-ordering rather than run-identity.

**[P1] Two boats in a single-width pound — concurrent runs on the same content-addressed OUTPUT_DIR deadlock/collide** — `skills/flux-engine/SKILL.md:112-117`, `skills/flux-engine/phases/launch.md:11-16`, `skills/flux-engine/SKILL.md:67`

The named mechanism: a single-width pound (the canal stretch between two locks) admits only one boat at a time in a given direction. If two boats enter from opposite ends, neither can pass and neither can reverse — a hard deadlock the keeper prevents by *scheduling*: boats are batched into same-direction cycles and a pound is never entered until confirmed clear.

The structural parallel: OUTPUT_DIR is content-addressed by `sha256(INPUT_PATH)[:8]` (SKILL.md:112-117, launch.md:11). This is *deliberately stable* — for prompt-cache hits — which means **two concurrent reviews of the same target resolve to the identical pound**. launch.md:13-16 then *deletes* all `*.md`, `*.md.partial`, `peer-findings.jsonl`, `decisions.log` before dispatch. Two boats, one pound: run B's pre-dispatch `find ... -delete` wipes run A's in-flight `.partial` files, and both write `.md` files for the same agent names into the same directory — directly violating the "at most one terminal `.md` per agent per run" invariant (`shared-contracts.md:76`), which is only enforced *within* a run, not across runs.

The insight/risk: the FLUX_RUN_UUID quire-mark (`launch.md:17-24`) is the mitigation — it lets synthesis *detect* cross-run contamination after the fact — but detection is the keeper noticing the collision after both boats are stuck, not preventing entry. The actual prevention exists only as opt-in: `--output-dir` override (SKILL.md:67) "for genuinely concurrent runs ... where the default content-addressed path would race." This makes safe concurrency a thing the *caller must remember*, and `/flux-review` (which fans out multiple tracks, each its own flux-drive run) does not appear to pass distinct OUTPUT_DIRs per track. Recommend a lock-keeper-style entry guard: an atomic lockfile (`{OUTPUT_DIR}/.run-{UUID}.lock`) created *before* the destructive `find -delete`; if a foreign live lock exists, the entering run must either wait (single-direction batching) or auto-suffix its own UUID into the path — turning a silent collision into an explicit "pound occupied" wait.

**[P1] Lockful drawdown vs. budget — the concurrency cap (6) is a fixed lock size, not sized to the water budget** — `skills/flux-engine/phases/launch.md:160-174`, `config/flux-drive/budget.yaml` (no `dispatch:` key)

The named mechanism: a well-run flight sizes its batching cycle to the *available summit water*, not to a fixed number. In a drought the keeper passes boats in smaller batches (less water per cycle); in flood, larger. A lock whose batch size is fixed regardless of reservoir level will either waste water (over-batch in drought → run dry) or waste throughput (under-batch in flood).

The structural parallel: `launch.md:161` sets `MAX_CONCURRENT_AGENTS = ${MAX_CONCURRENT_AGENTS:-6}` and the resolution order at line 164 is "env var → budget.yaml `dispatch.max_concurrent_agents` → default 6." **But `config/flux-drive/budget.yaml` has no `dispatch:` section at all** (verified — keys are budgets, agent_defaults, dropout, incremental_expansion, cross_model_dispatch, etc.). So the documented middle tier of the resolution chain is dead: the cap is *always* either an env override or the hardcoded 6. Meanwhile budget.yaml is rich with *water-budget* data — per-input token budgets (`budgets:`), per-agent token estimates (`agent_defaults:`), `enforcement: soft` — none of which feeds the concurrency cap. The lock size (6) and the water budget (token budgets) are decoupled.

The insight/risk: the "Why 6" rationale (launch.md:171) is an Erlang-C argument about the *summit pound* (API rate limit), so the cap *should* be derived from current reservoir pressure, yet it is a constant. A 40k-token-budget brainstorm and a 300k-token-budget repo review both fan out at 6. Two fixes, increasing in ambition: (a) add the missing `dispatch.max_concurrent_agents` key to budget.yaml so the documented resolution order is real, not phantom; (b) better, derive the cap from budget pressure — when `BUDGET_TOTAL` is small or `enforcement` is biting, narrow the batch (drought), and widen it for large `repo` budgets (flood). Today the batching cycle is arbitrary with respect to the water it consumes.

**[P2] Fill/empty time — per-stage latency is governed by the slowest chamber, and stall-rescue is the keeper's drain-relief valve** — `skills/flux-engine/phases/launch.md:262-270`, `shared-contracts.md:79-81`

The named mechanism: a staircase descends only as fast as its slowest chamber fills. The keeper cannot start the next equalization until the current lock reaches level; a jammed paddle (a chamber that won't fill) stalls every boat behind it. The traditional relief is a *bypass weir / overflow* — a controlled drain that lets the flight keep moving water past a stuck chamber rather than backing up the whole pound.

The structural parallel: flux-watch (launch.md:262-270) waits up to the full 300s Task timeout × N agents in the worst case; a single stalled agent in `timeout_original_running` (`shared-contracts.md:79`) holds synthesis. `STALL_RESCUE=1` (launch.md:268, shared-contracts.md:81) is precisely the bypass weir: after `STALL_TIMEOUT` (60s) of no overall progress, it writes a stall stub (verdict `error`, `kind:stall`) so "synthesis treats the stall as data, not silence," saving "up to 16 minutes of wall-clock per stalled agent." This is exactly the keeper opening the overflow so the flight keeps moving.

The insight/risk: the bypass weir is **off by default** ("Off by default for back-compat"). A lock-keeper would never run a flight with the overflow valves disabled — a single jam strands every boat. The default posture optimizes for the rare case where a slow-but-eventually-successful agent is preferable to a stall stub, at the cost of the common case where one genuinely-hung agent serializes the whole run to the full timeout. Recommend flipping the default to `STALL_RESCUE=1` (the spec already calls partial-progress synthesis "preferable to all-or-nothing" for "any review"), with the strict all-or-nothing mode as the opt-in. The water-budget framing makes the trade-off stark: keeping the flight moving past one jam is almost always worth the lost lockful.

**[P2] The keeper's batch is per-lock, but the flight needs a flight-wide schedule — track caps don't compose into a global plan** — `skills/flux-review-engine/phases/track-dispatch.md:189-196`, `skills/flux-review-engine/SKILL.md:144-153`

The named mechanism: on a multi-lock flight, each lock-keeper batching their own boats is *necessary but not sufficient* — without a flight-wide schedule, a downstream keeper can fill from the summit at the same moment an upstream keeper does, double-drawing the reservoir even though each keeper individually respects their own chamber's limits. Famous flights post a single master timetable; per-lock discretion is bounded by it.

The structural parallel: track-dispatch.md:189-196 nests two independent caps — `MAX_CONCURRENT_TRACKS` (outer) and `PER_TRACK_AGENT_CAP` (inner, passed as each flux-drive's `MAX_CONCURRENT_AGENTS`). Each track is "its own keeper." But there is no flight-wide accounting object: the outer wrapper multiplies caps to *estimate* a peak (12) but does not maintain a shared live counter that all tracks decrement against. The comment at launch.md:170 admits this — "The cap is per flux-drive run. Outer wrappers like /flux-review apply their own per-track cap *on top*." Caps stack multiplicatively rather than reconciling against one budget.

The insight/risk: this is the same reservoir as P0 but at the scheduling layer rather than the count layer. The robustness story (SKILL.md:153, "If any track fails, the command degrades gracefully") is good per-keeper discipline, but graceful degradation of *throughput* is not the same as preventing *summit overdraw*. A master-timetable fix: have `/flux-review` create one shared semaphore file (keyed by FLUX_RUN_UUID family), and have every inner flux-drive acquire from it rather than from a private per-track counter — making the per-track cap a *ceiling* under a real flight-wide budget, the way a posted lock timetable bounds each keeper's discretion.
