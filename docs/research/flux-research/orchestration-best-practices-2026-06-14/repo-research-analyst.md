### Findings

**Fan-out / dispatch.** Two distinct fan-out engines exist.
- *flux-engine (single flux-drive run):* agents are dispatched as parallel `Task` calls with `run_in_background: true`. In review mode dispatch is **staged** — Stage 1 launches the top 2-3 scored agents (launch.md:176-186), Stage 2 expands conditionally (expansion.md:305). In research mode all agents are dispatched at once in Step 2.1-research (launch.md:52-56). Agent types: Plugin agents use native `subagent_type` (e.g. `interflux:review:fd-architecture`); Project agents use `subagent_type: general-purpose` with the agent file pasted as system prompt; Cross-AI/Oracle runs via Bash background (launch.md:238-253). Documents are written once to a temp file and Read by agents to avoid prompt duplication (launch.md:68-108).
- *flux-review-engine (multi-track):* a two-phase fan-out — Phase/Step 2 fans out N tracks in parallel, each generating a specialized agent set; Step 3 fans out one independent flux-drive review per track in parallel (track-dispatch.md:1-5, 185-234). Tracks are at increasing semantic distance (adjacent → orthogonal → distant → esoteric), triaged 2-4 by target characteristics (SKILL.md:69-88).

**Concurrency / rate-limit control.** Cap is `MAX_CONCURRENT_AGENTS` (default 6), resolved env → `budget.yaml dispatch.max_concurrent_agents` → 6 (launch.md:160-164). The dispatch loop tracks in-flight background Tasks and blocks before issuing the next dispatch once the cap is hit, waiting for any in-flight Task to complete (launch.md:166-172). Rationale is explicitly Erlang-C queueing: "Erlang C predicts ~30% retry-token waste at the 16-agent fan-out tier" and "6 concurrent agents stays inside the per-minute concurrent-session limit while keeping Stage 1 (2-3 agents) unconstrained" (launch.md:158, 172). The cap is per flux-drive run and applies to Stage 1, Stage 2 expansion, research dispatch, and peer-finding broadcasts. For multi-track, an outer/inner cap pair prevents multiplicative blow-up: `MAX_CONCURRENT_TRACKS=4` × `PER_TRACK_AGENT_CAP=3` = 12 concurrent agents (each track's inner flux-drive receives `MAX_CONCURRENT_AGENTS=3`) (track-dispatch.md:185-196). Stage 2 dispatches in batches when candidate count exceeds the cap (expansion.md:305).

**Staged dispatch.** Review mode is two-stage by design (staging.md:13-31). Scoring assigns the top 40% of slots (min 2, max 5) to Stage 1, the rest to Stage 2; agents scoring ≥2 but unselected form an expansion pool (scoring.md:168-193). Between stages: optional synchronous research dispatch (max 2, 60s each) enriches Stage 2 context (staging.md:36-59); AgentDropout prunes redundant Stage 2 candidates by a 0.6 redundancy score, never dropping exempt fd-safety/fd-correctness (staging.md:165-197); an expansion decision uses severity×adjacency scoring (P0 in adjacent domain +3, P1 +2, disagreement +2, domain affinity +1) with thresholds ≥3 recommend / 2 offer / ≤1 stop, and **always** requires user approval — never auto-expands (staging.md:82-163). Research mode is single-stage (all agents at once, launch.md:52-56). Multi-track flux-review is staged at a coarser grain (generate-then-review across tracks).

**Completion detection.** Filesystem-signal based, not return-value based. Agents write `{agent}.md.partial` while working and perform an atomic `.partial → .md` rename as the final action — the rename is the binding completion signal; the `<!-- flux-drive:complete -->` sentinel is diagnostic metadata only (shared-contracts.md:25-32; completion-signal.md:11-26). The orchestrator detects completion by `.md` presence. Monitoring is via `scripts/flux-watch.sh {OUTPUT_DIR} {N} {TIMEOUT}` using inotifywait, falling back to 5s polling; timeouts 300s (Task) / 600s (Codex); research mode uses depth timeouts (quick 30s / standard 2min / deep 5min) and the `<!-- flux-research:complete -->` sentinel (launch.md:262-266; shared-contracts.md:186-201). A full Dispatch State Machine documents states dispatched → writing → completed / timeout_original_running → retried (shared-contracts.md:35-72).

**Partial-failure handling.** Multiple layered mechanisms:
- *Invariant:* every launched agent ends with exactly one `.md` (findings, error stub, or stall stub), so synthesis always processes exactly N files (completion-signal.md:58-68; shared-contracts.md:74-81).
- *Retry Race Protocol:* on `.partial` with no `.md`, touch `{agent}.abort`, rename the partial to `.aborted-<epoch>` so the slow original's late `mv` fails harmlessly, then synchronous retry (`run_in_background: false`, 300s); error stub if retry fails (launch.md:270; shared-contracts.md:83-111).
- *Stall rescue (opt-in):* `STALL_RESCUE=1` writes a `verdict: error` stall stub + `kind:stall` peer-finding when an expected agent emits nothing within `STALL_TIMEOUT` (60s) of progress, saving up to 16 min wall-clock (launch.md:268).
- *Synthetic refusal detection:* if a subagent transcript ends with the deterministic Usage-Policy refusal string, write `{agent}.refused.md` and auto-retry once at a **downgraded model tier** (opus→sonnet→haiku); never retry same tier (launch.md:272-307).
- *Foreign-file detection:* synthesis skips outputs whose first line doesn't match the run's `<!-- run-uuid -->` quire-mark, preventing stale/concurrent-run contamination (launch.md:17-24; synthesize.md:27-32).
- *Synthesis treatment:* error/refused/stall stubs are counted but contribute zero findings and don't count toward convergence (shared-contracts.md:80-81; synthesize.md:38-39). Multi-track and per-track generation degrade gracefully — if a track fails, surviving tracks proceed (track-dispatch.md:183; SKILL.md:153).

**Synthesis (fan-in).** The host orchestrator never reads individual agent files; it delegates the entire fan-in to an external **intersynth** subagent — `intersynth:synthesize-review` (review) or `intersynth:synthesize-research` (research) — which collects, validates, deduplicates, tracks convergence, validates discourse moves, writes verdict, and returns a compact ~10-15-line summary plus `summary.md` + `findings.json` (synthesize.md:45-108). Collection is index-first/two-tier: parse the `### Findings Index` (~30 lines) per agent, read prose only on demand/conflict (synthesis.md:32-52). Five dedup rules: same file:line+same issue → merge & credit all; same file:line+diff issue → co-located; same issue+diff location → cross-reference; conflicting severity → highest; conflicting recommendation → preserve both (synthesize.md:110-119; synthesis.md:54-80). Verdict is deterministic: any P0 → risky, any P1 → needs-changes, else safe (synthesize.md:218). Convergence (N agents agreeing) is the confidence signal; for multi-track, cross-track convergence (same issue from independent semantic-distance reasoning paths) is ranked highest and tracks are merged by a separate cross-track synthesis subagent (track-synthesis.md:1-100). Outputs: `summary.md`, `findings.json` (with cost_report, dropout, reaction sub-objects), individual `{agent}.md` reports; review mode optionally creates beads (synthesize.md:180-214, 445-501).

**Triage / scoring.** Pre-filter (keyword filters for file/dir, routing-pattern matching for diffs) reduces the candidate pool before scoring; base_score=0 agents are always excluded (scoring.md:85-111). Score = `base_score(0-3) + domain_boost(0-2, binary on injection-criteria presence) + project_bonus(0-1) + domain_agent_bonus(0-1)`, then `× trust_multiplier(0.05-1.0)` from interspect feedback; max 7 (scoring.md:13-139). Dynamic slot ceiling = base 4 + scope (0-3 by single-file/small-diff/large-diff/repo) + domain (0-2 by detected domain count), hard max 10 (scoring.md:142-161). Stage assignment: top 40% → Stage 1 (min 2, max 5), remainder → Stage 2, ≥2 unselected → expansion pool; tiebreak project > plugin > cross-ai (scoring.md:168-193). "Affinity" appears as (a) the domain adjacency map in staging.md:62-80 (architecture↔[performance,quality], correctness↔[safety,performance], etc., intentionally asymmetric) driving expansion + AgentDropout, and (b) domain detection signals in config/flux-drive/domains/index.yaml mapping directories/files/frameworks to domain profiles whose injection criteria drive domain_boost. Empirical validation (staging.md:197): across 120+ runs all P0/P1 findings came from 6 plugin agents; project-generated agents produced 0% P0/P1.

### Confidence

High. Every mechanism is documented in dedicated, internally consistent spec + skill files with concrete parameter values, algorithms, and worked examples, and the two engines (flux-engine, flux-review-engine) cross-reference each other coherently (e.g. `PER_TRACK_AGENT_CAP` passed as inner `MAX_CONCURRENT_AGENTS`). Supporting scripts referenced (`flux-watch.sh`, `findings-helper.sh`) exist on disk, raising confidence the docs reflect a real implementation rather than aspiration. Confidence is High for "what the protocol specifies"; it is necessarily lower for "what executes at runtime," since the orchestration is prose-instruction-driven (a host LLM follows launch.md/synthesize.md) rather than a deterministic program, so adherence depends on the host model following the steps.

### Gaps

- **No runtime scheduler.** The concurrency loop, retry race, and staging are LLM-followed prose, not enforced code. There is no semaphore/work-queue primitive guaranteeing the cap; "track in-flight Task calls" is an instruction to the host (launch.md:166-172). Drift between spec and execution is possible and unmeasured here.
- **Polling vs event detection divergence.** The spec contract says SHOULD poll every 30s (completion-signal.md:103); the implementation prefers inotifywait with 5s polling fallback (shared-contracts.md:186-201). The 30s figure is not what flux-watch uses.
- **Erlang-C numbers are unsourced heuristics.** The "~30% retry-token waste at 16 agents" and "why 6" claims (launch.md:158-172) have no cited measurement or model parameters in-repo; they read as design rationale, not validated telemetry.
- **Trust multiplier / interspect, qmd knowledge, Composer, FluxBench challenger, intersynth, lib-routing** are all external progressive enhancements; their internals are out of this repo. If absent, behavior silently falls back (e.g. trust=1.0, frontmatter models). Exact fallback fidelity is not specified end-to-end.
- **Dedup fuzzy-match thresholds** (Levenshtein <0.3 or 3+ shared keywords, synthesis.md:60) are executed inside the intersynth subagent's context — there is no deterministic implementation in this repo to inspect; quality depends on that agent.
- **Adjacency map is hardcoded and unlearned** (staging.md:274) and omits the 5 cognitive agents (fd-systems, fd-decisions, fd-people, fd-resilience, fd-perception) — adjacency/expansion is specified only for the 7 technical agents plus game-design.
- **Cross-track convergence scoring** is described qualitatively (rank by N tracks agreeing) but has no numeric formula or dedup-rule equivalent to the single-run 5 rules (track-synthesis.md:36-43).
- **No backpressure on the synthesis subagent** itself, and no specified behavior if intersynth is unavailable (synthesize.md assumes it launches).
